#52026-03-14T10:58:30.993Z

hmm now it's the right url but i get a 426 upgrade required

That's Apache rejecting the WebSocket upgrade — it's not forwarding the Upgrade header through to nginx. You need mod_proxy_wstunnel enabled and a separate ProxyPass rule for the WebSocket path.

In your Apache VirtualHost config, alongside whatever proxies /shift-tac-toe/ to your container, add this before the main ProxyPass (more specific paths must come first):

# WebSocket — must be before the regular ProxyPass for /shift-tac-toe/
ProxyPass /shift-tac-toe/ws ws://localhost:5780/ws
ProxyPassReverse /shift-tac-toe/ws ws://localhost:5780/ws

Replace 5780 with whatever port your container is on. The ws:// scheme is what tells Apache to use mod_proxy_wstunnel instead of the regular HTTP proxy.

If the module isn't loaded yet:

sudo a2enmod proxy_wstunnel
sudo systemctl reload apache2

The path-stripping chain (Apache → nginx → gameserver) is a lot of moving parts for WebSockets. A subdomain is genuinely simpler here — wss://ws.blahaj.beauty gets its own clean VirtualHost and bypasses all the path gymnastics.

The changes would be:

docker-compose.yml — expose the gameserver port to the host:

gameserver:
  ports:
    - "8080:8080"   # instead of expose

Apache — new VirtualHost for the subdomain (with Let's Encrypt cert):

<VirtualHost *:443>
  ServerName ws.blahaj.beauty
  SSLEngine on
  # ... your cert directives ...

  ProxyPreserveHost On
  ProxyPass / ws://localhost:8080/
  ProxyPassReverse / ws://localhost:8080/
</VirtualHost>

Client — instead of deriving the WS URL from BASE_URL, use a Vite env variable:

In .env.production:

VITE_WS_URL=wss://ws.blahaj.beauty

Then in useMultiplayer.ts:

const wsUrl = import.meta.env.VITE_WS_URL
  ?? `${window.location.protocol === 'https:' ? 'wss:' : 'ws:'}//${window.location.host}${import.meta.env.BASE_URL}ws`;
const ws = new WebSocket(wsUrl);

This keeps dev working as-is (falls back to the Vite proxy) while production connects to the subdomain directly.
