To build and run locally:

To deploy to your server:

# Option A — push to a registry, pull on server
docker build -t shift-tac-toe .
docker tag shift-tac-toe yourregistry/shift-tac-toe:latest
docker push yourregistry/shift-tac-toe:latest
# on server:
docker pull yourregistry/shift-tac-toe:latest
docker run -d -p 80:80 --restart unless-stopped yourregistry/shift-tac-toe:latest

# Option B — build directly on the server (if you have git there)
git push && ssh yourserver "cd shift-tac-toe && git pull && docker build -t shift-tac-toe . && docker run -d -p 80:80 --restart unless-stopped shift-tac-toe"

If you want to run it on a non-standard port (e.g. behind a reverse proxy like Caddy/nginx on the host), just change the -p flag — e.g. -p 3000:80.

#22026-03-14T01:20:02.226Z

hmm, the thing is i'll already have an apache server running on the host which i'm planning to set up to forward to this. do you think it makes sense to run an nginx here anyway? i guess that's fine

Yeah, it's totally fine and actually the standard pattern — the inner nginx is just a lightweight static file server (~25MB alpine image), and Apache doesn't care what's behind the proxy. You'd just configure Apache something like:

<VirtualHost *:443>
    ServerName shift-tac-toe.yourdomain.com
    ProxyPass / http://localhost:8080/
    ProxyPassReverse / http://localhost:8080/
</VirtualHost>

The alternative would be ditching nginx and using vite preview or a Node static server, but those are worse choices — nginx is faster, handles caching headers properly, and the image stays small. Double-proxying adds essentially zero overhead for a game with a handful of concurrent players.
